## 1. Назначение документа

Настоящее техническое задание описывает требования к разработке web-first/PWA приложения для управления личными, семейными, рабочими и общественными задачами с поддержкой AI-классификации, матрицы Эйзенхауэра, напоминаний, утреннего дашборда и вечернего отчета.

Документ предназначен для:

1. постановки задач в Cursor;
2. проверки реализации через Codex;
3. контроля архитектурных решений;
4. формирования MVP;
5. последующей подготовки технического стека и плана разработки;
6. будущей адаптации приложения под Android.

## 2. Общая идея продукта

Продукт представляет собой self-hosted web-first систему управления задачами для одного основного владельца, членов семьи и ограниченного числа рабочих партнеров.

Основная задача системы — быстро превращать хаотичный поток входящих дел, мыслей, поручений и обязательств в структурированные задачи с категориями, сроками, напоминаниями, приоритетами и правами доступа.

Приложение должно помогать пользователю:

1. быстро фиксировать задачи голосом или текстом;
2. не забывать задачи;
3. видеть список задач на день утром;
4. вечером видеть, что было выполнено;
5. переносить, удалять или делегировать невыполненные задачи;
6. раскладывать задачи по матрице Эйзенхауэра;
7. отделять личные, семейные, рабочие и партнерские задачи;
8. давать доступ другим пользователям только к нужным задачам;
9. в будущем связывать задачи с долгосрочной стратегией, целями и проектами.

## 5. Пользователи и роли

### 5.1 Типы пользователей

В первой версии предполагается до 10 пользователей:

1. владелец системы;
2. члены семьи;
3. рабочие партнеры;
4. временные участники;
5. гости.

### 5.2 Роли

#### Owner

Владелец системы.

Права:

1. управляет всеми системными настройками;
2. создает и удаляет пользователей;
3. создает пространства;
4. назначает роли;
5. видит системную аналитику;
6. управляет резервным копированием;
7. управляет AI-настройками;
8. управляет глобальными категориями.

Ограничение:

Owner не должен автоматически получать доступ к личным задачам других пользователей, если архитектурно выбран режим строгой приватности.

#### Admin

Администратор пространства.

Права:

1. управляет пользователями внутри конкретного пространства;
2. управляет проектами внутри пространства;
3. управляет задачами, доступными этому пространству;
4. назначает исполнителей;
5. просматривает аналитику пространства.

Ограничение:

Admin пространства не видит личные задачи пользователя вне данного пространства.

#### Member

Обычный участник.

Права:

1. видит задачи доступных ему пространств;
2. создает задачи в разрешенных пространствах;
3. редактирует свои задачи;
4. отмечает задачи выполненными;
5. комментирует задачи;
6. получает уведомления.

#### Guest

Гость.

Права:

1. видит только явно выданные задачи или проекты;
2. может комментировать, если разрешено;
3. может выполнять назначенные задачи, если разрешено.

#### Private User

Пользователь личного пространства.

Права:

1. видит только свои личные задачи;
2. управляет своими задачами;
3. использует AI-классификацию;
4. получает личные напоминания.

## 8. Функциональные требования

## 8.1 Авторизация и пользователи

### FR-AUTH-001

Система должна поддерживать регистрацию и вход пользователей.

Минимум для MVP:

1. email;
2. пароль;
3. сессия пользователя;
4. logout;
5. восстановление пароля можно перенести на этап 1.1.

### FR-AUTH-002

Система должна поддерживать роли:

1. Owner;
2. Admin;
3. Member;
4. Guest.

### FR-AUTH-003

Система должна ограничивать доступ к данным на уровне API, а не только на уровне UI.

Критерий приемки:

Пользователь A не может получить задачу пользователя B прямым API-запросом, если у него нет прав.

### FR-AUTH-004

Система должна поддерживать приглашение пользователя в пространство.

Минимум MVP:

1. Owner/Admin создает пользователя вручную;
2. пользователь получает временный пароль или ссылку приглашения;
3. приглашенный пользователь видит только доступные пространства.

## 8.2 Пространства и права доступа

### FR-SPACE-001

Система должна позволять создавать пространства.

Обязательные типы:

1. Личное;
2. Семья;
3. Работа;
4. Партнеры;
5. Общественное;
6. Входящие.

### FR-SPACE-002

Каждая задача должна принадлежать одному пространству.

### FR-SPACE-003

Пользователь видит задачи пространства только если он является участником пространства или имеет прямой доступ к задаче.

### FR-SPACE-004

Личное пространство пользователя недоступно другим пользователям по умолчанию.

### FR-SPACE-005

Администратор одного пространства не получает автоматический доступ к другим пространствам.

Редактируемые поля:

1. title;
2. description;
3. status;
4. due_at;
5. scheduled_for;

7. category_id;
8. project_id;
9. importance_score;
10. urgency_score;
11. eisenhower_quadrant;
12. tags;

### FR-TASK-003

Система должна поддерживать удаление задачи через soft delete.

Задача не удаляется физически из базы, а получает deleted_at.

### FR-TASK-004

Система должна поддерживать выполнение задачи.

При выполнении:

1. status меняется на done;
2. заполняется completed_at;
3. создается событие task_completed.

### FR-TASK-005

Система должна поддерживать перенос задачи.

При переносе:

1. меняется scheduled_for или due_at;
2. создается событие task_rescheduled;
3. при необходимости пересчитывается срочность.

### FR-TASK-006

Система должна поддерживать подзадачи.

Подзадача имеет parent_task_id.

### FR-TASK-007

Система должна поддерживать статус waiting.

Статус используется для задач, которые ожидают ответа другого человека или внешнего действия.

## 8.4 Быстрый ввод задачи

### FR-CAPTURE-001

Система должна иметь кнопку “Быстро добавить”.

### FR-CAPTURE-002

Быстрое добавление должно позволять создать задачу только по тексту.

Пример:


### FR-CAPTURE-003

После быстрого ввода задача должна пройти AI-классификацию, если AI включен.

### FR-CAPTURE-004

Если AI не уверен, задача должна попадать во “Входящие”.

## 8.5 Голосовой ввод

### FR-VOICE-001

Система должна поддерживать запись голосовой задачи через браузер.

### FR-VOICE-002

Пользователь должен иметь возможность нажать кнопку микрофона, надиктовать задачу и отправить запись на обработку.

### FR-VOICE-003

Голосовая запись должна преобразовываться в текст через speech-to-text слой.

### FR-VOICE-004

После распознавания текст должен передаваться в AI-классификатор.

### FR-VOICE-005

Пользователь должен видеть распознанный текст перед окончательным сохранением, если включен режим подтверждения.

### FR-VOICE-006

Если speech-to-text недоступен, система должна позволять создать задачу вручную.

## 8.6 AI-классификация задач

### FR-AI-001

Система должна принимать текст задачи и возвращать структурированный JSON.

Вход:

```json
{
  "text": "Завтра утром позвонить заказчику по лендингу, это важно, напомни за час",
  "user_id": "current_user",
  "timezone": "Europe/Moscow"
}
```

Выход:

```json
{
```

### FR-AI-002

AI не должен иметь право сам предоставлять доступ другим пользователям.

### FR-AI-005

AI не должен автоматически переносить задачу в семейное или рабочее пространство, если есть риск раскрытия приватной информации.

При исправлении создается событие:

```
ai_classification_corrected
```

## 8.7 Матрица Эйзенхауэра

### FR-EIS-001

Каждая задача должна иметь поля:

1. importance_score;
2. urgency_score;
3. eisenhower_quadrant.

### FR-EIS-002

Значения importance_score и urgency_score должны быть от 1 до 5.

### FR-EIS-003

Квадранты:

1. important_urgent;
2. important_not_urgent;
3. not_important_urgent;
4. not_important_not_urgent.

### FR-EIS-004

Правила расчета по умолчанию:

```
importance_score >= 4 и urgency_score >= 4 → important_urgent
importance_score >= 4 и urgency_score < 4 → important_not_urgent
importance_score < 4 и urgency_score >= 4 → not_important_urgent
importance_score < 4 и urgency_score < 4 → not_important_not_urgent
```

### FR-EIS-005

Пользователь должен иметь возможность вручную изменить квадрат.

### FR-EIS-006

Изменение квадранта должно логироваться в TaskEvent.

### FR-EIS-007

Должен быть отдельный экран матрицы Эйзенхауэра.

## 10. Нефункциональные требования

## 10.1 Производительность

### NFR-PERF-001

Основные экраны должны открываться быстро на малом объеме данных.

Целевые значения MVP:

1. открытие Today dashboard: до 1 секунды на 1000 задач;
2. поиск задач: до 1 секунды на 1000 задач;
3. создание задачи: до 500 мс без AI;
4. создание задачи с AI: допустимо дольше, но UI не должен зависать.

### NFR-PERF-002

AI-классификация должна выполняться асинхронно или с понятным индикатором загрузки.

### NFR-PERF-003

Worker напоминаний не должен блокировать основной интерфейс.

## 10.2 Безопасность

### NFR-SEC-001

API-ключи внешних сервисов не должны попадать во frontend.

### NFR-SEC-002

Все действия с задачами должны проверять права доступа на backend.

### NFR-SEC-003

Пароли должны храниться только в виде хеша.

### NFR-SEC-004

Должна быть защита от доступа к чужим задачам через прямые API-запросы.

### NFR-SEC-005

Должна быть валидация входных данных.

### NFR-SEC-006

Должны быть системные логи ошибок.

### NFR-SEC-007

AI-запросы не должны отправлять больше данных, чем нужно для классификации конкретной задачи.

## 10.3 Приватность

### NFR-PRIV-001

Личные задачи пользователя являются приватными по умолчанию.

### NFR-PRIV-002

Аналитика не должна раскрывать содержимое чужих приватных задач.

### NFR-PRIV-003

AI-логи должны быть доступны только владельцу задачи и системному Owner в режиме технического аудита без раскрытия лишнего контента, если выбран строгий режим приватности.

### NFR-PRIV-004

При удалении пользователя его задачи должны либо архивироваться, либо передаваться владельцу workspace, либо удаляться по явному сценарию.

## 10.4 Надежность

### NFR-REL-001

Приложение должно работать на сервере 24/7.

### NFR-REL-002

Система должна поддерживать backup базы данных.

### NFR-REL-003

Система должна корректно восстанавливаться после перезапуска.

### NFR-REL-004

Напоминания не должны теряться при перезапуске worker.

### NFR-REL-005

Повторяющиеся задачи не должны создаваться дублями.

## 10.5 Наблюдаемость

### NFR-OBS-001

Система должна логировать ошибки backend.

### NFR-OBS-002

Система должна логировать ошибки AI-классификации.

### NFR-OBS-003

Система должна логировать работу worker напоминаний.

### NFR-OBS-004

В будущем должна быть страница технического health-check.

Минимальные проверки:

1. приложение работает;
2. база доступна;
3. worker работает;
4. очередь доступна;
5. AI provider доступен;
6. последнее успешное напоминание;
7. последнее успешное резервное копирование.

## 10.6 Тестируемость

### NFR-TEST-001

В проекте должны быть unit-тесты для доменной логики.

### NFR-TEST-002

Должны быть integration-тесты для API.

### NFR-TEST-003

Должны быть e2e-тесты для основных пользовательских сценариев.

### NFR-TEST-004

Обязательные тестовые сценарии:

1. User A не видит private task User B.
2. Member семьи не видит рабочие задачи.
3. Партнер не видит семейные задачи.
4. Задача создается через quick add.
5. Задача классифицируется AI.
6. Задача с низким confidence попадает во “Входящие”.
7. Задача отображается в Today dashboard.
8. Выполненная задача попадает в вечерний отчет.
9. Перенос задачи создает TaskEvent.
10. Напоминание не отправляется дважды.
11. Повторяющаяся задача создает следующий экземпляр.
12. Soft deleted задача не отображается в списках.

## 13. Сценарии использования

### UC-001. Быстро добавить задачу текстом

1. Пользователь открывает Today dashboard.
2. Нажимает “Быстро добавить”.
3. Вводит текст задачи.
4. Система отправляет текст в AI-классификатор.
5. AI возвращает JSON.
6. Система создает задачу.
7. Если confidence высокий — задача попадает в нужное пространство.
8. Если confidence низкий — задача попадает во “Входящие”.

Критерий приемки:

Задача создана, отображается в списке, имеет категорию, статус и событие task_created.

### UC-002. Добавить задачу голосом

1. Пользователь нажимает микрофон.
2. Надиктовывает задачу.
3. Система записывает аудио.
4. Аудио преобразуется в текст.
5. Текст отображается пользователю.
6. Система классифицирует задачу.
7. Пользователь подтверждает или редактирует.
8. Задача сохраняется.

Критерий приемки:

Задача создана из голосового ввода, а распознанный текст сохранен в AI log.

### UC-003. Утреннее планирование

1. Пользователь утром открывает приложение.
2. Система показывает Today dashboard.
3. Пользователь видит просроченные задачи.
4. Пользователь видит задачи на сегодня.
5. Пользователь выбирает главные задачи.
6. Пользователь выполняет или переносит задачи.

Критерий приемки:

Все задачи на сегодня видны, просроченные вынесены наверх.

### UC-004. Вечерний разбор

1. Пользователь открывает “Итоги дня”.
2. Система показывает выполненные и невыполненные задачи.
3. Пользователь принимает решение по каждой невыполненной задаче.
4. Система логирует действия.
5. Система показывает итог дня.

Критерий приемки:

Ни одна задача дня не остается без статуса разбора, если пользователь завершил вечерний отчет.

### UC-005. Семейная задача

1. Пользователь создает задачу в пространстве “Семья”.
2. Назначает члена семьи.
3. Член семьи видит задачу.
4. Рабочий партнер не видит задачу.

Критерий приемки:

Права доступа работают корректно.

### UC-006. Рабочая задача для партнера

1. Пользователь создает рабочую задачу.
2. Назначает партнера исполнителем.
3. Партнер видит только эту задачу или проект.
4. Партнер не видит личные и семейные задачи.

Критерий приемки:

Приватность не нарушена.

## 14. Архитектурные требования

### 14.1 Общий стиль

Рекомендуемый стиль:

1. modular monolith для MVP;
2. четкое разделение frontend/backend/domain/worker;
3. отдельный слой AI;
4. отдельный слой access control;
5. отдельный слой событий;
6. отдельный worker для фоновых задач.

### 14.2 Слои

Система должна быть разделена на слои:

1. UI Layer;
2. API Layer;
3. Domain Layer;
4. Access Control Layer;
5. Persistence Layer;
6. AI Layer;
7. Worker Layer;
8. Notification Layer;
9. Observability Layer.

### 14.3 Доменная логика

Доменная логика не должна быть размазана по UI-компонентам.

Правила:

1. расчет матрицы Эйзенхауэра — domain service;

3. создание событий — event service;
4. обработка напоминаний — worker/service;
5. AI-классификация — отдельный AI service.

### 14.4 Фоновые задачи

Фоновые задачи:

1. отправка напоминаний;
2. генерация повторяющихся задач;
3. вечерние системные проверки;
4. будущие digest-уведомления;
5. будущая AI-пакетная обработка.

## 25. Итоговое определение MVP

MVP — это web-first self-hosted приложение для 1–10 пользователей, которое позволяет быстро создавать задачи, распределять их по личным/семейным/рабочим пространствам, автоматически классифицировать через AI, раскладывать по матрице Эйзенхауэра, получать напоминания, видеть план дня утром и итоги вечером, при этом строго соблюдая приватность и фиксируя все ключевые изменения в журнале событий.
